## CRITICAL FINDINGS

### CR-1: SHA-256 vs argon2id API Key Hashing Contradiction

**Lines:** 2192 vs 3583, 3605, 3446

- Line 2192 (API Authentication §): "Raw key never stored — only SHA-256 hash in `forge.api_keys`."
- Line 3583 (Security §): "argon2id (`argon2-cffi>=23.1.0`). Plaintext never stored."
- Line 3446 (Decisions Locked): "Customer API keys: argon2id hashed"

**Impact:** A developer reading Thread 7's API Authentication section implements SHA-256. A developer reading Infrastructure implements argon2id. These are fundamentally different algorithms. No gate criterion catches the mismatch.

**Fix:** Change line 2192 to argon2id. Add Phase 5 gate criterion: "API key hash algorithm = argon2id verified."

### CR-6: Phase 3 Gate Missing `marts.signals_history` Criterion (Consolidated Plan Locked This)

**Lines:** v3.1 3226–3233 vs consolidated plan §3 Phase 3

- Consolidated plan explicitly locked: "`marts.signals_history` populated with `computed_at`, `ingested_at`, `regime`, `pillar_scores`, `p_bullish` fields"
- This is labeled "explicit gate criterion" because the B3 performance endpoint depends on `ingested_at` being present in signal records from Phase 3 onward

**Fix:** Add two rows to Phase 3 gate table: (1) `marts.signals_history` field completeness, (2) regime stored at emission time.

## HIGH FINDINGS

### H-1: `forge_compute` — No Operational Specification

**Lines:** 95, 1702, 3568, 3705, 3727–3735

ADR-006 calls it a "Python service." ADR-004 says DuckDB "runs embedded in forge_compute." It appears in the credential model (`gold_reader` mounted on `forge_compute`). But:
- No Docker service definition
- No Dockerfile
- No entry point
- No trigger mechanism (Dagster asset? cron? manual?)
- Not in the service table (lines 3563–3570)

A developer cannot determine whether to build a Docker container, a Dagster op, or a Python package.

**Fix:** Add operational spec: is it a Dagster asset group inside `empire_dagster_code`? If so, the `gold_reader` credential mounts on that container.

## MEDIUM FINDINGS

### M-1: `forge.current_values` DDL Missing
No CREATE TABLE/MATERIALIZED VIEW for the ClickHouse `current_values` view. CLAUDE.md describes it as AggregatingMergeTree with argMaxState but no DDL in the design doc.

### M-2: Dead Letter 90-Day TTL Missing From DDL
CLAUDE.md says TTL 90 days. v3.1 DDL (lines 1577–1594) has no TTL clause. Table would grow indefinitely.

### M-3: `signal_eligible` Flag — No DDL Home
Line 1653 references `signal_eligible = false` for `defi.lending.reward_apy`. No column for this exists in the metrics table DDL.

### M-4: CFTC COT `ingested_at` Assignment Not Explicit
Lines 2002–2004 specify `observed_at` = Tuesday but don't state `ingested_at` = Friday (actual collection). Misinterpretation breaks PIT.

### M-5: No Phase 1 Gate for `ingested_at` Correctness
The entire PIT system depends on `ingested_at` = actual wall-clock ingestion time. No Phase 1 gate criterion verifies this.

### M-6: No argon2id Gate Criterion
No Phase 5 or 6 gate verifies the hashing algorithm. Given CR-1's contradiction, this gap is especially dangerous.

### M-7: No Neutral Threshold ±0.10 Gate
No gate in Phase 3, 5, or 6 verifies the threshold. Incorrect threshold silently corrupts performance metrics.

### M-8: `signal_snapshot_writer` Missing From Dagster Asset Dependency Graph
Lines 2691–2703 show the dependency graph but omit `signal_snapshot_writer`. A developer doesn't know what it depends on.

### M-9: `forge_compute` Startup Check Unspecified
Line 3732 references a "startup check" but doesn't specify behavior on failure.

### M-10: Great Expectations — No Specification
Phase 1 gate requires GE but: no suite names, no expectation definitions, no failure behavior specified.

### M-11: `bronze-hot-lifecycle.xml` Not Provided
Line 3828 references it in deployment runbook. File doesn't exist in repo.

### M-12: `pillar_attribution` Scoping Undefined
B3 field-tier mapping says Protocol gets `pillar_attribution` "scoped" — scoped to what?

### M-13: Socrata Pagination Unresolved
Line 2009 flags it but no resolution. No gate verifies CFTC COT historical backfill.

### M-14: Phase 6 Missing "First Customer Delivery" Gate
Consolidated plan requires it. v3.1 mentions it in steps but has no gate row.

### M-15: `marts.signals_history` — dbt Model or Something Else?
Referenced as a "Phase 3 output" in the Dagster graph but not in the explicit dbt model list. Implementation type unspecified.
